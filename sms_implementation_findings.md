# SMS Implementation Findings

## Overview
The current SMS implementation in the Unified Notifications system utilizes **Lancola SMS** as the primary provider. The system is designed with a multi-tenant architecture, allowing each organization to use either global default credentials or their own specific provider details.

---

## 1. Primary Provider: Lancola SMS
- **Provider URL:** `https://sms.lancolatech.com/api/services/sendsms/`
- **Default Shortcode:** "Maziwa Tele"
- **Authentication:** Uses a combination of `apikey`, `partnerID`, and `shortcode`.

## 2. Technical Implementation
The implementation is centralized in the `backend/src/integrations/lancola-sms` directory:

- **`LancolaSmsService`:** This NestJS service (`lancola-sms.service.ts`) is responsible for dispatching messages. It retrieves organization-specific credentials and falls back to environment variables if they are missing.
- **`prepareMessage` Helper:** A utility function in `lancola-sms.functions.ts` that constructs the final API request URL with encoded parameters.
- **Normalization:** Phone numbers are automatically normalized to include the country code (defaulting to `254` for Kenya) before being sent.
- **Cost & Credits:** Each message has a "rate" (default is 1 credit per SMS) which is deducted from the organization's credit balance upon successful dispatch.

## 3. Organization Setup (Multi-tenancy)
Organization-specific settings are stored in the `Organization` MongoDB collection.

### Credential Precedence
When sending an SMS, the system looks for credentials in the following order:
1.  **Organization Document:** `org.credentials.sms_apiKey`, `org.credentials.sms_apiUrl`, etc.
2.  **Environment Variables:** `LANCOLA_SMS_apiKey`, `LANCOLA_SMS_APIURL`, etc.
3.  **Schema Defaults:** Hardcoded defaults in the `OrganizationSchema`.

### Configuration Fields
The following fields can be configured per organization:
- `LANCOLA_SMS_APIURL`
- `LANCOLA_SMS_apiKey`
- `LANCOLA_SMS_partnerID`
- `LANCOLA_SMS_shortCode`

## 4. Observations & Future Changes
- **Belio SMS Integration:** Documentation exists (`Belio_SMS_Intergration.md`) for a future integration with **Belio Bulk SMS**. This new provider uses OAuth2 Client Credentials Flow but is **not yet active** in the core codebase.
- **Fallback Mechanism:** While the architecture supports per-org credentials, there is currently no automatic failover between providers (e.g., if Lancola is down, it doesn't automatically switch to another provider unless manually reconfigured).
